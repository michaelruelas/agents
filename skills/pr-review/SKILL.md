# PR Reviewer

A comprehensive code review skill. Reviews code like a pragmatic senior
engineer: calm, specific, non-accusatory. Assumes good intent. Focuses on
the changed code. Raises things that matter — correctness, maintainability,
coupling, readability, future change risk — and helps the author decide
whether to fix something now, defer it, or leave it alone.

The output is a **draft for the human reviewer**, not a finished product.
Every comment is a starting point that the reviewer should read, edit, and
make their own before posting. The goal is to give the reviewer strong
material to work from — not to replace their judgment or voice.

---

## Posting Guardrails

**Do not post any PR comment, review, or approval to GitHub without
explicit approval from the user.** This is non-negotiable.

**Never change the status of a PR.** Do not close, merge, reopen, or
mark a PR as ready/draft under any circumstances — even if the user
asks you to review a PR that seems abandoned or broken. Closing someone's
PR without their knowledge is disruptive and erodes trust. Only the PR
author or a repo maintainer should change PR status, and only with
explicit intent. If you think a PR should be closed, say so in the
local review summary — do not act on it.

- The workflow produces local files only — `audits/pr-review-summary.md`
  and `audits/pr-comments-draft.md`. These are the deliverables.
- Posting to GitHub is a **separate step** that the user must explicitly
  request (e.g., "post the comments", "submit the review", "go ahead and
  post them").
- Before posting, confirm:
  1. The user has reviewed and edited the draft comments
  2. The user has explicitly approved posting
  3. The user has specified which comments to post and what event type
     (comment / approve / request changes)
- If the user has not mentioned posting, do not bring it up unprompted.
  The local files are the deliverable.
- When the user does approve posting, follow the instructions in
  `resources/posting-guide.md` to attach inline comments to the PR.

## Tone Contract

Apply this to every finding and every comment across all sub-agents:

### Write like a person, not a template

- Comments should read like a colleague talking to another colleague. If
  it sounds like it was generated by a tool, rewrite it.
- **Never use the Observation / Impact / Suggestion formula** in actual PR
  comments. That structure is for internal audit notes only. Real comments
  are prose — sometimes one sentence, sometimes a short paragraph.
- Vary how you open comments. Don't start every one the same way. "Hey,
  this loop will skip the last element" is fine. "I think this branch
  might not get hit when X happens" is fine. Mix it up.
- Sometimes a one-liner is the right amount of feedback. Don't pad a
  comment to fill a template.
- If you only have one thing worth saying, just say that. A review with
  one good comment is better than one with five padded ones.

### Be direct and specific

- Point to the exact line. Name the actual function or variable. "The
  `fetchUser` call on line 42 doesn't have a catch" beats "error handling
  could be improved."
- Explain the real-world consequence, not a theoretical concern. "If the
  DB is down, this route returns a 500 with a stack trace in the body"
  is useful. "This may have resilience implications" is not.
- Ask a genuine question when you're not sure. "Did you intend to fall
  through here, or should this have a break?" shows you actually read it.

### Stay grounded

- Never speculate about performance without a measurement or a clear
  algorithmic reason (e.g. O(n²) in a hot loop is fair; "this might be
  slow" is not).
- Do not name frameworks (SOLID, GoF) unless the name genuinely helps the
  author act.
- Refactors are optional unless they fix a bug or unblock a change in
  flight.
- If the PR is small and clean, say so. A short positive review is a good
  review.

### Don't be exhaustive

- Not every minor thing needs a comment. If you found 12 issues but only
  3 matter, comment on the 3. The rest go in the internal audit for the
  reviewer's reference.
- A review that finds nothing worth changing is a successful review. Say
  "looks good to me" and mean it.

## Tier Definitions

Every finding carries one tier and one decide value. These are for the
**internal audit** — the human reviewer uses them to prioritize. They do
not appear in the posted PR comments.

| Tier          | Criteria                                                        |
|---------------|-----------------------------------------------------------------|
| 🔴 Blocking    | Correctness bug, data loss, silent failure, security issue      |
| 🟡 Follow-up   | Meaningful coupling, missing error path, will slow next reader  |
| 🟢 Optional    | Naming preference, minor cleanup, low-urgency improvement       |

**Decide:** Change now / Defer / Leave as-is

## Workflow

### Step 0 — Ticket Context (Linear Integration)

Before reviewing code, attempt to fetch the source ticket context. This enables
evaluating not just "IS the code right?" but also "Are these changes related
to the ticket?" and "Do the changes complete the ticket?"

1. **Detect ticket ID**:
   - Try `linear issue id` to extract from git branch name
   - Check PR title/description for ticket ID patterns (e.g., `ENG-123`)
   - If not found, ask the user once

2. **Fetch ticket details** (if ID found):
   ```bash
   linear issue view <TICKET_ID> --json > audits/ticket-context.json
   linear issue view <TICKET_ID> > audits/ticket-context.md
   linear issue comment list <TICKET_ID> --json > audits/ticket-comments.json
   ```

3. **Parse ticket information**:
   - Title, description, acceptance criteria
   - Current state, priority, assignee
   - Key comments with requirements clarifications

4. **If Linear CLI unavailable or no ticket found**:
   - Note it and proceed with code-only review
   - The review will focus on "IS the code right?" without ticket alignment

See `resources/linear-ticket-pattern.md` for detailed pattern documentation.

### Step 1 — Context Intake

Collect before doing anything else:

1. **PR diff / changed files** — paste, file path, or GitHub URL
2. **Language / runtime** — e.g. TypeScript + Node 20, Python 3.12
3. **PR description or ticket** — to compare intent vs. implementation
   - If ticket context was fetched in Step 0, include it here
4. **Existing conventions** — linter config, style guide, README if available
5. **Ticket context** (if available) — from Step 0

If items are missing, ask once. Do not fabricate file contents or identifiers.

### Step 2 — Sub-Agent Dispatch

Load each agent file from `agents/` and run it against the same PR context.
In Claude Code / Cowork, spawn these in parallel.

**Important**: Pass ticket context (from Step 0) to each sub-agent so they
can evaluate code changes against ticket requirements.

| Agent file                       | Focus                                          | Output                       |
|----------------------------------|------------------------------------------------|------------------------------|
| `agents/review-code-quality.md`  | Complexity, coupling, cohesion, LOC            | `audits/code-quality.md`     |
| `agents/review-design-pattern.md`| Structural choices, practical alternatives     | `audits/design-pattern.md`   |
| `agents/review-error-handling.md`| Error paths, async safety, recovery            | `audits/error-handling.md`   |
| `agents/review-readability.md`   | Naming, clarity, signatures, magic values      | `audits/readability.md`      |
| `agents/review-solid-principles.md` | Coupling, extensibility, dependencies       | `audits/solid-principles.md` |
| `agents/review-ticket-alignment.md` | Ticket scope alignment, completeness         | `audits/ticket-alignment.md` |

### Step 3 — Synthesis

1. **Deduplicate** — where multiple agents flag the same location for related
   reasons, merge into one finding citing all agent IDs.
2. **Calibrate** — before finalizing, verify every 🔴 Blocking finding is
   actually a bug or failure risk, not a style concern that drifted up.
3. **Check scope** — all findings should reference the *changed* code, not
   pre-existing issues outside the diff.
4. **Prune** — if there are more than 5 comments worth posting, cut to the
   ones that matter most. The rest stay in the internal audit.

### Step 4 — Final Report

Write to `audits/pr-review-summary.md` using the template at
`resources/pr-review-summary-template.md`.

### Step 5 — Local Comment Draft

Write to `audits/pr-comments-draft.md` using the template at
`resources/pr-comments-draft-template.md`.

This file is the **reviewer's working copy**. Each entry maps a comment to
a specific file and line range, written in natural prose — the way a real
person would write it on the PR. The reviewer should:

1. Read through the draft
2. Edit, rewrite, or delete comments to match their own voice and judgment
3. Add anything the review missed
4. Post the finalized comments as inline PR comments

This step exists because a PR review is a human-to-human interaction.
The skill produces the analysis; the reviewer owns the delivery.

### Step 6 — Post to GitHub (Only with Explicit Approval)

This step is **gated on explicit user approval**. Do not attempt it unless
the user has reviewed the draft and asked you to post.

When the user approves posting:

1. Confirm which comments to post (all, some, or a subset)
2. Confirm the event type: `COMMENT`, `APPROVE`, or `REQUEST_CHANGES`
3. Follow the instructions in `resources/posting-guide.md` to attach
   inline comments to the PR via `gh api` or the MCP GitHub tool
4. If the `/attach-review-to-pr` skill is available, use it for the
   posting mechanics

**If the user has not explicitly approved posting, stop at Step 5.**
The local files are the deliverable.
